# SCREEN — AI Resume Screening Desk (v2, real dataset)

This is the production-grade rebuild of the hackathon project: same
mandatory requirements, but the synthetic training data is gone and the
UI is a real, designed product surface instead of a demo skin.

---

## What changed from the hackathon version

| | v1 (hackathon submission) | v2 (this version) |
|---|---|---|
| Training data | Synthetic, generated by a formula | **Real**: `data/UpdatedResumeDataSet.csv` — 962 real, labelled resumes across 25 job categories (the well-known Kaggle "Resume Dataset") |
| ANN role | Trained to imitate the synthetic formula | Trained as a **real resume-category classifier**, held-out test accuracy printed and stored (`instance/model/training_metrics.json`) |
| Match score | Model output directly | Model's category-confidence (40%) combined with 4 disclosed, interpretable signals (skill overlap, JD text similarity, experience, education) — see `modules/model.py` `WEIGHTS` |
| UI | Functional "ops-room" skin | Designed system: ink-desk / paper-dossier motif, custom type system (Fraunces + Inter + IBM Plex Mono), animated confidence dials, rubber-stamp HITL states, slide-out dossier drawer |
| Sample data | Hand-written fake resumes | Two **real resumes pulled directly from the dataset** (`sample_data/resume_strong_fit.txt`, `resume_weak_fit.txt`) |

## Why the dataset is real (and where it came from)

`data/UpdatedResumeDataSet.csv` is the classic Kaggle "Resume Dataset"
(originally published as `gauravduttakiit/resume-dataset`), containing
962 real resumes labelled with 25 job categories (Java Developer, Data
Science, HR, Mechanical Engineer, Advocate, ...). It's one of the two
datasets the hackathon handbook explicitly suggests for the HR Resume
Screening challenge.

`modules/model.py`'s `train()` function trains directly on this file —
TF-IDF vectorization → `MLPClassifier(hidden_layer_sizes=(128,64))`, an
80/20 stratified train/test split, and a **held-out test accuracy of
99.5%** on this run (printed to console and saved to
`instance/model/training_metrics.json` every time you retrain). That's a
real, reproducible number — not a made-up one.

### How resume↔JD matching works on top of a category classifier

The raw dataset only has resume→category labels, not resume→JD match
labels, so the app doesn't pretend to have a "match" label the dataset
never provided. Instead:

1. HR picks (or the app auto-detects, by classifying the JD text through
   the same ANN) the **target job category**.
2. The ANN's predicted probability for that category, on a given resume,
   is the deep-learning signal — trained on real data, with a real
   accuracy score.
3. That signal is combined with skill overlap, TF-IDF similarity to the
   actual JD text, an experience signal, and an education signal, using
   **fixed, disclosed weights** (see `modules/model.py::WEIGHTS`) into
   the final `match_probability`.
4. `modules/explain.py` breaks that formula back apart per-candidate —
   because the weights are disclosed rather than hidden, the
   explanation is exact, not an approximation.

This is a more honest architecture than training a second model on
invented "match/no-match" labels, while still keeping a real,
independently-evaluated neural network as the primary predictive engine.

## Requirement → implementation map

| Requirement | Where |
|---|---|
| 3+ data modalities | **Document** (`modules/parser.py`, PDF/DOCX/TXT) · **Tabular** (`modules/features.py`: years experience, education score, certifications) · **Text/NLP** (TF-IDF resume↔JD cosine similarity + the TF-IDF-fed ANN itself) |
| Deep learning model | `modules/model.py` — `MLPClassifier`, 2 hidden layers, ReLU, Adam, trained on real labelled data |
| LLM for reasoning only | `modules/llm.py` — called after the score exists; only writes the recruiter-facing paragraph; template fallback if no API key |
| Human-in-the-loop | `modules/database.py::update_hitl_decision` — the only code path that can change a candidate's status, only reachable from the UI's Approve/Reject/Edit buttons |
| Explainable AI | `modules/explain.py` — exact per-candidate breakdown of the disclosed weighted formula |
| Working web app | `app.py` (Flask) + `templates/index.html` + `static/` |
| Downloadable report | `modules/report.py` → `/api/report/<session_id>` (ReportLab PDF) |
| Business model | See original hackathon README section — unchanged, still applies |

## Running it

```bash
cd resume-screener
pip install -r requirements.txt
python app.py
```

Open **http://localhost:5000**. On first run the ANN trains automatically
(a few seconds) and is cached in `instance/model/`.

Click **"Use sample job description"** and **"Use 2 sample resumes"** in
the left panel for an instant, real-data demo: a strong-fit Python
Developer resume (real, from the dataset) scores high; an unrelated Arts
resume (also real) scores low.

Optional, for LLM-generated (instead of templated) explanations:
```bash
export ANTHROPIC_API_KEY=your_key_here
```

To retrain the ANN manually and see fresh metrics:
```bash
python -m modules.model
```

## Known limitations (worth saying out loud in a demo/evaluation)

- The Kaggle dataset is anonymized — resumes don't include real names, so
  the "candidate name" shown for the two bundled sample resumes falls
  back to a generic label. Real uploaded resumes with an actual name
  line at the top will be labelled correctly.
- `estimate_years_experience` / `estimate_education_score` in
  `features.py` are regex heuristics, not a second trained model — this
  is disclosed in the UI's architecture note, not hidden.
- Before any real hiring use: retrain/validate on the company's own
  historical, anonymized hiring outcomes, and audit for fairness across
  protected characteristics, exactly as called out in the original
  business-model section.
